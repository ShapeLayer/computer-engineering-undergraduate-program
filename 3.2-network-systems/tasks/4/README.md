# 第04回 ネットワークシステム 課題

課題(〆 10/17 17:00)
本日の授業で取り上げたシングルサインオン認証について、この(shibboleth認証以外も含む)の概要やメリット/デメリットについて簡単(400字程度)にまとめなさい。
　　※ 10/31(金) 17:00 締め切り

SSO는 다양한 서비스에 걸쳐서 일관된 사용자 신원 인증을 구현할 수 있다는 데에 큰 장점이 있다. 특히 최근의 환경은 매우 다양한 유형의 아키텍쳐가 혼재되어 있고, 같이 독립적인 개별 소프트웨어 여러 개를 합해 운용하게 되는 경우가 많다. 이와 같은 상황에서 개별 시스템마다 독립적인 신원 인증의 운용은 개발 코스트, 시스템 유지보수 측면 뿐 아니라 UX에도 좋다고 할 수 없다. SSO는 이들 시스템 사이에 신원 인증을 공유함으로써, 작업 소요를 낮추고, 각종 보안 대책 수립이나 업데이트 작업에서 집중적으로 시스템을 관리할 수 있게 한다. 사용자에게는 1회의 신원 인증으로 여러 서비스를 자연스럽게 넘나들며 사용할 수 있으므로 사용 편의성이 증가한다.
하지만 이는 개별 신원 인증에 비해 공격에 성공하였을 때의 이익이 높아지게 되어 더욱 많은 공격의 표적이 될 수 있다. SSO 처리는 개별 시스템 외부에서 처리되는 경우가 많으므로, 역시 신원 인증을 위한 통신 절차 등은 구현해야 하는데, 이 과정에서 MSA의 문제점과 같이 예측하지 못한 오류를 연이어 겪으며 개발 비용이 오히려 높아지거나 새로운 공격 벡터를 형성하게 될 수 있다.

SSOは、様々なサービスにわたって一貫した認証を実装できるという点に大きなメリットがあります。最近の環境では、多様なアーキテクチャが混在し、複数のSWを合わせて運用するケースが多くなっています。このような状況において、システムごとに独立した認証を運用することは、開発コストやシステム保守の側面だけでなく、UXの観点からも望ましいとは言えません。SSOはこれらのシステム間で認証を共有することにより、作業工数を削減し、各種セキュリティ対策の策定やアップデート作業において、システムを集中的に管理することができます。ユーザーにとっては、1回の認証で複数のサービスを自然に行き来しながら利用できるため、利便性が向上します。
しかし、これは個別の認証に比べて攻撃に成功した際の利益が大きくなるため、より多くの攻撃の標的となり得ます。SSOの処理は個別のシステムの外部で行われることが多いため、認証のための通信手順などを別途実装する必要があります。その過程で、MSAの問題点のように予期せぬエラーが多発し、かえって開発コストが増大したり、新たな攻撃ベクトルを生み出してしまったりする可能性もあります。
